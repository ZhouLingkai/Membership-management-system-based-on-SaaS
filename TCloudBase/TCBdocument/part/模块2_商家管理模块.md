# 模块2：商家管理模块

> 云函数路径前缀：`/api/v1/merchant/`
> 所有接口均需先调用 `HttpUtils.parseRequest(event)` 解析入参。
> 响应统一格式：`{ code, data, message }`
> **手机号安全规范**：前端使用 AES-256-CBC 加密传输，云函数收到后立即 `decryptPhone()` 解密，以 **11 位明文**写入数据库。所有涉及手机号的参数均传 AES 密文。

---

## 接口列表

| # | 接口名称 | 路径 | HTTP 方法 | 认证 |
|---|---|---|---|---|
| 1 | 获取验证码 | `/api/v1/merchant/verifyCode` | POST | 无 |
| 2 | 注册/登录 | `/api/v1/merchant/login` | POST | 无（type=1/2）；login令牌①（type=3） |

## 任务a：`verify_codes` 集合结构

```js
{
  _id: String,
  phone: String,          // 手机号（明文存储，传输时加密）
  code: String,           // 6位数字验证码（明文）
  codeExpireTime: Date,   // createTime + 5分钟（验证码过期时间，新请求时更新）
  used: Boolean,          // 单次使用，使用后置 true，防重复消费
  dailyCount: Number,     // 当天已发送次数（上限5次）
  lastSendTime: Date,     // 最近成功发送时间（1分钟频率限制基准）
  expireTime: Date        // 次日零点（文档清理时间）
}
```

**索引：** `phone`（查询指定手机号记录）、`expireTime`（定时清理，每天凌晨）

> **设计说明**：
> - 每日首次请求：集合中无该 `phone` 记录，直接 `add` 新增
> - 非首次请求：按 `phone` 查询已有记录
>   - `now < lastSendTime + 1min`：返回频率限制，附带剩余等待秒数
>   - `dailyCount >= 5`：返回今日次数已达上限
>   - 否则：更新 `code`、`codeExpireTime`、`lastSendTime`，`dailyCount++`，重置 `used = false`
> - 验证时比对 `code`、`codeExpireTime`、`used`，验证成功后将 `used` 置 `true`
> - `expireTime` 为次日零点，定时任务凌晨清理

---

## 任务b：`login_fail_records` 集合结构

```js
{
  _id: String,
  role: String,         // 'merchant' | 'staff' | 'member'
  identifier: String,   // 锁定标识（唯一索引）：merchant=手机号明文；staff=账号名；member=手机号明文
  failCount: Number,    // 当前连续失败次数
  lockUntil: Date,      // 锁定到期时间，null 表示未锁定
  lastFailTime: Date,
  updateTime: Date
}
```

**索引：** `identifier`（唯一索引）、清理频率：次日凌晨

**梯度锁定策略：**

| 连续失败次数 | 锁定时长 |
|---|---|
| 5 次 | 15 分钟 |
| 再失败 5 次（共10次） | 60 分钟 |
| 再失败 5 次（共15次） | 24 小时 |

> 登录成功后直接删除该 `identifier` 对应记录

## 任务c：`token_blacklist` 集合结构

（已在模块1中定义，此处复述）

```js
{
  _id: String,
  jti: String,          // JWT jti 字段，令牌唯一标识（唯一索引）
  expireTime: Date      // 与原令牌 exp 一致，过期后文档无意义
}
```

**使用场景：**
- 退出登录：将①②③的 jti 写入黑名单
- 改密：tokenVersion 自增（全局失效①）+ 当前设备②③写黑名单
- 令牌刷新：旧令牌 jti 写黑名单

## 接口1：获取验证码

- **URL**：`/api/v1/merchant/verifyCode`
- **HTTP Method**：POST
- **认证**：无

### 请求参数（body）

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `phone` | String | 是 | 手机号（AES 密文，云函数内调用 `decryptPhone` 解密） |
| `type` | Number | 是 | 1=注册验证码 2=找回密码验证码 |

### 核心逻辑

1. `decryptPhone(phone)` 解密手机号
2. `validatePhone(phone)` 校验手机号格式
3. 查询 `merchants` 集合，判断手机号是否已注册：
   - `type=1`（注册码）：手机号**未注册**才发，已注册返回错误
   - `type=2`（找回码）：手机号**已注册**才发，未注册返回错误
4. 查询 `verify_codes`，按 `phone` 查找记录：
   - 记录不存在（今日首次）：直接进入步骤5
   - `now < lastSendTime + 1min`：拒绝，返回剩余等待秒数
   - `dailyCount >= 5`：今日已达上限，拒绝
5. 生成验证码：**当前日期 6 位**，格式 `YYMMDD`（如 260414）
6. 写入 `verify_codes`：
   - 首次（无记录）：`add`，`dailyCount=1`，`expireTime=tomorrowMidnight()`
   - 非首次：更新 `code`、`codeExpireTime=now+5min`、`lastSendTime=now`、`dailyCount++`、`used=false`
7. 返回成功（不返回验证码）

> **Mock阶段**：跳过真实短信发送，验证码固定为当天日期（`YYMMDD` 格式，如 `260414`），需通过其他渠道告知测试人员；生产环境调用真实短信接口，`data` 中不返回验证码字段。

### 响应示例

```json
{
  "code": 200,
  "data": {},
  "message": "验证码已发送"
}
```

### 错误码

| code | ErrorCodes 键 | 场景 |
|---|---|---|
| 90002 | `PARAM_MISSING` | 缺少必要参数 phone 或 type |
| 90003 | `PHONE_FORMAT_INVALID` | 手机号格式错误 |
| 20004 | `VERIFY_CODE_WRONG_TYPE` | type=1 但手机号已注册 / type=2 但手机号未注册 |
| 20002 | `VERIFY_CODE_SEND_TOO_FAST` | 1 分钟内已发送，附带 `{ remainSeconds: N }` |
| 20003 | `VERIFY_CODE_DAILY_LIMIT` | 今日发送次数已达上限（5次） |

## 接口2：注册/登录（多功能接口）

- **URL**：`/api/v1/merchant/login`
- **HTTP Method**：POST
- **认证**：type=3 时需携带商家登录令牌①

### 请求参数（body）

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | Number | 是 | 1=注册 2=账密登录 3=自动登录（令牌换取） |
| `phone` | String | type=1/2 | 手机号 AES 密文 |
| `password` | String | type=1/2 | 登录密码（AES 密文，云函数内 `decryptAES` 解密后 Argon2 验证/哈希） |
| `verifyCode` | String | type=1 | 注册时的短信验证码 |
| `deviceId` | String | 是 | 设备唯一标识（写入令牌 `device_id`） |

> type=3 时，令牌从 `Authorization` Header 读取，body 只需 `{ type: 3, deviceId }`。

### 核心逻辑

#### type=1：注册

1. `decryptPhone` + `validatePhone` 校验手机号
2. `decryptAES(password)` 解密密码，`validatePassword` 校验密码强度
3. 查询 `merchants`，手机号已存在则拒绝
4. 查询 `verify_codes`，验证码匹配（`code`相等）且 `codeExpireTime > now` 且 `used === false`，验证成功后将 `used` 置 `true`
5. `hashPassword(password)` 哈希密码
6. `decryptPhone` 解密后以明文存入 `merchants.phone`
7. 插入 `merchants` 文档（`_id = generateUUID()`，`tokenVersion = 0`）
8. 签发①商家登录令牌（7d）+ ②商家访问令牌（4h）
9. 返回两个令牌 + 商家基础信息

#### type=2：账密登录

1. `decryptPhone` + `validatePhone` 校验手机号
2. `decryptAES(password)` 解密密码
3. 查询 `login_fail_records`（`identifier=手机号明文，role='merchant'`），检查是否被锁定（`lockUntil > now`）
4. 查询 `merchants`（`where({ phone: 手机号明文 })`），不存在返回 404
5. `verifyPassword(password, merchant.password)` 验证密码：
   - **失败**：在 `login_fail_records` 中 upsert，`failCount++`，按梯度更新 `lockUntil`，返回剩余次数
   - **成功**：删除 `login_fail_records` 对应记录
6. 签发①（7d）+ ②（4h）
7. 返回两个令牌 + 商家基础信息

#### type=3：自动登录

1. `verifyToken(loginToken)` 验证商家登录令牌①
2. 校验 `payload.role === 'merchant'` && `payload.type === 'login'`
3. 查询 `token_blacklist`，jti 在黑名单则拒绝（401）
4. 查询 `merchants`，校验 `tokenVersion === payload.v`（版本一致则登录令牌未被全局失效）
5. 签发新②商家访问令牌（4h）
6. 返回新访问令牌 + 商家基础信息

### 响应结果（type=1/2 统一格式）

| 字段 | 类型 | 说明 |
|---|---|---|
| `data.loginToken` | String | `Bearer <商家登录令牌①>`（7d） |
| `data.loginTokenExpireTime` | Number | 登录令牌过期 Unix 毫秒时间戳（7d后） |
| `data.accessToken` | String | `Bearer <商家访问令牌②>`（4h） |
| `data.accessTokenExpireTime` | Number | 访问令牌过期 Unix 毫秒时间戳（4h后） |
| `data.merchantInfo` | Object | `{ merchantId, merchantName, avatar, merchantLevel }` |

### 响应结果（type=3）

| 字段 | 类型 | 说明 |
|---|---|---|
| `data.accessToken` | String | `Bearer <商家访问令牌②>`（4h） |
| `data.accessTokenExpireTime` | Number | 访问令牌过期 Unix 毫秒时间戳（4h后） |
| `data.merchantInfo` | Object | 同上 |

### 响应示例（type=2）

```json
{
  "code": 200,
  "data": {
    "loginToken": "Bearer eyJhbGci...",
    "loginTokenExpireTime": 1745286000000,
    "accessToken": "Bearer eyJhbGci...",
    "accessTokenExpireTime": 1744682400000,

      "merchantId": "uuid-merchant-xxx",

}
```

| 90003 | `PHONE_FORMAT_INVALID` | 手机号格式错误（type=1/2） |

## 接口5：修改密码

- **URL**：`/api/v1/merchant/password/change`
- **HTTP Method**：POST
- **认证**：商家访问令牌②

### 请求参数（body）

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `oldPassword` | String | 是 | 旧密码（AES 密文，云函数内 `decryptAES` 解密后验证） |
| `newPassword` | String | 是 | 新密码（AES 密文，云函数内 `decryptAES` 解密后哈希存储） |

### 核心逻辑

1. `verifyToken(accessToken)` 取 `merchantId`，层② 查黑名单
2. `decryptAES(oldPassword)`、`decryptAES(newPassword)` 解密两个密码
3. `validatePassword(newPassword)` 校验新密码强度
4. 查询 `merchants`，`verifyPassword(oldPassword, merchant.password)` 验证旧密码
5. `hashPassword(newPassword)` 哈希新密码
6. 更新 `merchants`：`password`、`tokenVersion + 1`（全局失效所有登录令牌①）、`updateTime`
7. 将当前访问令牌② jti 写入 `token_blacklist`（使本次 access 令牌立即失效）
8. 返回成功

> `tokenVersion + 1` 已使所有设备的登录令牌①全部失效，无需额外传入 loginJti。前端收到成功响应后应清除本地所有令牌，强制重新登录。

### 响应示例

```json
{
  "code": 200,
  "data": {},
  "message": "密码修改成功，请重新登录"
}
```

### 错误码

| code | ErrorCodes 键 | 场景 |
|---|---|---|
| 90002 | `PARAM_MISSING` | 缺少必要参数 |
| 90004 | `PASSWORD_WEAK` | 新密码强度不足 |
| 30003 | `MERCHANT_PASSWORD_WRONG` | 旧密码错误 |
| 10001 | `TOKEN_MISSING` | 访问令牌为空 |
| 10002 | `TOKEN_EXPIRED` | 访问令牌已过期 |
| 10003 | `TOKEN_BLACKLISTED` | 访问令牌已注销 |
| 10004 | `TOKEN_INSUFFICIENT_ROLE` | 令牌角色不符（非 merchant） |
| 10005 | `TOKEN_TYPE_MISMATCH` | 令牌类型不符（非 access） |

## 接口7：修改二级密码

- **URL**：`/api/v1/merchant/sndPswd/change`
- **HTTP Method**：POST
- **认证**：商家访问令牌②（需已登录，手机号从令牌 uid 查库获取，无需前端再传）
- **前置流程**：前端须先调用接口1（`verifyCode`，`type=2`）获取短信验证码

### 请求头

| 字段 | 必填 | 说明 |
|---|---|---|
| `Authorization` | 是 | `Bearer <商家访问令牌②>` |
| `Content-Type` | 是 | `application/json` |

### 请求参数（body）

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `verifyCode` | String | 是 | 短信验证码（type=2 发送的找回码，明文） |
| `newSndPswd` | String | 是 | 新二级密码（AES 密文，云函数内 `decryptAES` 解密后哈希存储） |

### 核心逻辑

1. `verifyToken(accessToken)` 取 `payload.uid`，层② 查黑名单
2. 校验 `payload.role === 'merchant'` && `payload.type === 'access'`
3. 校验 `verifyCode`、`newSndPswd` 参数不为空
4. `decryptAES(newSndPswd)` 解密新二级密码，`validatePassword` 校验密码强度
5. 查询 `merchants`（`_id === payload.uid`）获取手机号
6. 查询 `verify_codes`（`phone === merchant.phone`），校验验证码（`code` 相等且 `codeExpireTime > now` 且 `used === false`），成功后将 `used` 置 `true`
7. `hashPassword(newSndPswd)` 哈希新二级密码
8. 更新 `merchants`：`sndPswd`、`updateTime`

> 修改二级密码**不触发 tokenVersion 自增**，不影响已有登录令牌的有效性。

### 响应示例

```json
{
  "code": 200,
  "data": {},
  "message": "二级密码修改成功"
}
```

### 错误码

| code | ErrorCodes 键 | 场景 |
|---|---|---|
| 90002 | `PARAM_MISSING` | 缺少 verifyCode 或 newSndPswd |
| 90001 | `PARAM_INVALID` | newSndPswd 解密失败 |
| 90004 | `PASSWORD_WEAK` | 新二级密码强度不足 |
| 20001 | `VERIFY_CODE_INVALID` | 验证码错误、已过期或已使用 |
| 30002 | `MERCHANT_NOT_FOUND` | 商家不存在 |
| 10001 | `TOKEN_MISSING` | 访问令牌为空 |
| 10002 | `TOKEN_EXPIRED` | 访问令牌已过期 |
| 10003 | `TOKEN_BLACKLISTED` | 访问令牌已注销 |
