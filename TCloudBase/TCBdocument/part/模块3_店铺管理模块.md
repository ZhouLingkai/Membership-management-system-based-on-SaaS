# 模块3：店铺管理模块

> 云函数路径：`/api/v1/stores/`
> 所有接口均需先调用 `HttpUtils.parseRequest(event)` 解析入参。
> 响应统一格式：`{ code, data, message }`，`code=200` 为成功，其余为失败。

## 接口列表

| # | 接口名称 | 云函数路径建议 | HTTP 方法 | 功能 |
|---|---|---|---|---|
| 1 | 创建店铺 | `/api/v1/stores/create` | POST | 创建新店铺，校验商家等级+额外名额上限 |

## 集合依赖

| 集合 | 用途 |
|---|---|
| `merchants` | 查询商家等级（`merchantLevel`）与额外名额（`additionalStores`），计算开店上限 |
| `stores` | 店铺信息读写，统计当前店铺数量，跨店组管理 |

### 店铺数量上限规则

| merchantLevel | 基础名额（A） |
|---|---|
| 1（普通） | 1 |
| 2（VIP） | 3 |
| 3（SVIP） | 5 |

> **总上限 = A + B**，其中 B = `merchants.additionalStores`（商家单独购买的增量名额）。

### `shareGroup` 字段说明

- 同一商家名下，相同 `shareGroup` 值的店铺**共享会员数据**（会员可跨店消费/积分）。
- 独立经营的店铺拥有**唯一** `shareGroup` 值，与其他店铺相互隔离。
- 创建新店铺时，`shareGroup` 初始值 = 当前商家所有店铺 `shareGroup` 最大值 + 1。

## 接口1：创建店铺

- **URL**：`/api/v1/stores/create`
- **HTTP Method**：POST
- **认证**：商家访问令牌②（`role=merchant`，`type=access`）

### 请求头

| 字段 | 必填 | 说明 |
|---|---|---|
| `Authorization` | 是 | `Bearer <accessToken>` |

### 请求参数（body）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `storeName` | String | 是 | 店铺名称 |
| `storeType` | String | 否 | 店铺类型，不填为 null |
| `address` | String | 否 | 地址，不填为 null |

### 核心逻辑

1. 层① 无状态校验：验证访问令牌②签名与有效期。
2. 校验 `role=merchant`、`type=access`，否则返回 `TOKEN_INSUFFICIENT_ROLE` / `TOKEN_TYPE_MISMATCH`。
3. 层② 黑名单校验（写操作必查）。
4. 校验参数：`storeName` 不能为空。
5. 查 `merchants` 集合获取商家的 `merchantLevel` 和 `additionalStores`，计算开店上限 = 基础名额（A）+ `additionalStores`（B）。
6. 查 `stores` 集合统计当前商家（`merchantId = payload.uid`）名下店铺数量，若 **≥ 上限**则拒绝。
7. 查 `stores` 集合获取当前商家所有店铺的 `shareGroup` 最大值，新店铺 `shareGroup = max + 1`（若无店铺则从 1 开始）。
8. 向 `stores` 集合插入新文档，初始化所有字段默认值。
9. 返回新店铺 `storeId`。

### 响应结果

| 字段 | 类型 | 说明 |
|---|---|---|
| `data.storeId` | String | 新店铺 ID |

### 错误码

| code | ErrorCodes 键 | 场景 |
|---|---|---|
| 90002 | `PARAM_MISSING` | 缺少 storeName |
| 40003 | `STORE_LIMIT_EXCEEDED` | 已达该商家店铺数量上限 |
| 30002 | `MERCHANT_NOT_FOUND` | 商家账号不存在 |
| 10001 | `TOKEN_MISSING` | 令牌为空 |
| 10002 | `TOKEN_EXPIRED` | 令牌已过期 |
| 10003 | `TOKEN_BLACKLISTED` | 令牌已注销 |
| 10004 | `TOKEN_INSUFFICIENT_ROLE` | 非 merchant 角色 |
| 10005 | `TOKEN_TYPE_MISMATCH` | 非 access 类型令牌 |

- **URL**：`/api/v1/stores/info`

### 请求头

| 字段 | 必填 | 说明 |
|---|---|---|
| `Authorization` | 是 | `Bearer <accessToken>` |

### 请求参数（body）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|

1. 层① 无状态校验。

## 接口6：修改跨店组

- **URL**：`/api/v1/stores/shareGroup/update`
- **HTTP Method**：POST
- **认证**：商家访问令牌②

### 请求头

| 字段 | 必填 | 说明 |
|---|---|---|
| `Authorization` | 是 | `Bearer <accessToken>` |

### 请求参数（body）

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `type` | Number | 是 | 操作类型：1-新建组 2-加入组 3-移出组 4-解散组 |
| `storeIds` | Array\<String\> | type=1 时必填 | 新建组的店铺 ID 列表（≥2个，均须独立经营） |
| `storeId` | String | type=2/3 时必填 | 目标店铺 ID |
| `targetShareGroup` | Number | type=2 时必填 | 要加入的目标跨店组标识值 |
| `shareGroup` | Number | type=4 时必填 | 要解散的跨店组标识值 |

### 四种操作详解

#### type=1：新建跨店组

> 将 2 个或以上独立经营的店铺合并为一个跨店组。

**校验：**
- `storeIds` 长度 ≥ 2，所有店铺均属于当前商家。
- 所有店铺当前必须**独立经营**（`shareGroup` 值各不相同，且不与其他未选中店铺共用）。

**逻辑：**
- 取 `storeIds` 中最小的 `shareGroup` 值作为新组的组标识。
- 将 `storeIds` 内所有店铺的 `shareGroup` 设为该值。

#### type=2：加入已有跨店组

> 将一个独立店铺加入已有跨店组。

**校验：**
- `storeId` 存在且属于当前商家。
- 该店铺当前必须**独立经营**（其 `shareGroup` 唯一，不与其他店铺共用）。
- `targetShareGroup` 对应的跨店组在当前商家名下存在（至少有 ≥2 家店铺持有该值）。

**逻辑：**
- 将 `storeId` 的 `shareGroup` 设为 `targetShareGroup`。

#### type=3：移出跨店组

> 将某店铺从跨店组中单独拆出，成为独立经营。

**校验：**
- `storeId` 存在且属于当前商家。
- 该店铺当前必须处于某跨店组（`shareGroup` 值与至少另一家店铺相同）。

**逻辑：**
- 查当前商家所有店铺的 `shareGroup` 最大值 `maxGroup`。
- 将 `storeId` 的 `shareGroup` 设为 `maxGroup + 1`。

#### type=4：解散整个跨店组

> 将跨店组内所有店铺全部拆散，各自独立经营。

**校验：**
- `shareGroup` 对应的跨店组在当前商家名下存在（至少 ≥2 家店铺）。

**逻辑：**
- 查当前商家所有店铺 `shareGroup` 最大值 `maxGroup`。
- 将该组内每家店铺依次设为 `maxGroup + 1`，`maxGroup + 2`，…（每家唯一值）。

### 错误码

| code | ErrorCodes 键 | 场景 |
|---|---|---|
| 90002 | `PARAM_MISSING` | 缺少必要参数 |
| 90001 | `PARAM_INVALID` | type 非法；storeIds 少于 2 个；目标组不存在；店铺已在组中（type=2）；店铺不在任何组（type=3） |
| 40001 | `STORE_NOT_FOUND` | 店铺不存在 |
| 40002 | `STORE_NOT_BELONG` | 店铺不属于该商家 |
| 10001 | `TOKEN_MISSING` | 令牌为空 |
| 10002 | `TOKEN_EXPIRED` | 令牌已过期 |
| 10003 | `TOKEN_BLACKLISTED` | 令牌已注销 |
| 10004 | `TOKEN_INSUFFICIENT_ROLE` | 非 merchant 角色 |
| 10005 | `TOKEN_TYPE_MISMATCH` | 非 access 类型令牌 |

## 通用说明

### 令牌校验规范

所有接口均使用**商家访问令牌②**（`role=merchant`，`type=access`）。

| 层 | 触发 | 接口适用 |
|---|---|---|
| 层① 签名校验 | 所有接口 | 全部 |
| 层② 黑名单校验 | 写操作（创建/修改/跨店组变更） | 接口1/4/5/6 |
| 读操作跳过黑名单 | 查询性能优化 | 接口2/3 |

### 店铺归属校验

任何涉及 `storeId` 的操作，在查出 `stores` 文档后，必须验证 `doc.merchantId === payload.uid`，通过后方可继续。
