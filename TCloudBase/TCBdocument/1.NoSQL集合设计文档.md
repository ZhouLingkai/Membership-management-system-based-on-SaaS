# MySQL 数据表 → NoSQL 集合设计

> 设计原则：反范式化，允许冗余读字段，分次查询优先于联表，字段命名统一用 camelCase。

> **手机号安全规范**：手机号采用「加密传输、明文存储」策略。
> - **传输层**：前端（小程序/Web 端）统一使用 AES-256-CBC 加密后传输，防止明文在网络中暴露。
> - **存储层**：云函数收到后立即 `decryptPhone()` 解密，以 **11 位明文**存入数据库，保证查询效率（直接 `where({ phone })` 即可匹配，无随机 IV 问题）。
> - CloudBase 数据库有独立访问控制，外部无法直接读取集合，明文存储的风险在此架构下可接受。

## 1. `merchants` 集合

> 商家端独立账号体系，与会员端、员工端完全隔离。登录方式：手机号短信验证码注册 + 账密登录。移除跨店功能，一个商家账号下所有店铺，会员都可以消费/积分，如果相环境隔离开（比如只能在店A消费，不能再店B消费），必须注册多个账号。

```js
{
  _id: String,                  // 商家_ID，文档ID

  // ---- 账号信息（原 t_user） ----
  phone: String,                // 手机号（明文存储，传输时加密；唯一索引），必填
  password: String,             // 登录密码（Argon2 哈希），必填
  merchantName: String,         // 商家名称，默认 null
  avatar: String,               // 头像 URL，默认 null

  // ---- 商家扩展信息（原 t_merchant_extend） ----
  contactEmail: String,         // 联系邮箱，默认 null
  merchantIntro: String,        // 商家介绍，默认 null
  merchantLevel: Number,        // 1-普通 2-VIP 3-SVIP，默认为1
  sndPswd: String,              // 二级密码（Argon2 哈希，用于危险操作确认），默认与注册密码相同；修改须通过短信验证码验证与注册密码一样
  privilegeExpireTime: Date,    // 商家特权过期时间，默认为null
  additionalStores: Number,    // 最大店铺数量额外上限

  haveClient: Boolean,          // 是否有预期有自营客户端
  tokenVersion: Number,         // 登录令牌版本号（退出/改密自增）
  createTime: Date,             // 创建时间
  updateTime: Date,             // 更新时间
  extJson: Object               // 扩展字段
}
```
**后续拓展字段**:
会员上限

**索引字段：**
- `phone`（唯一索引，登录查询）
- `merchantLevel`（筛选等级）

## 2. `stores` 集合

> 本系统仅支持连锁模式，会员由商家端创建，要么会员在所有店铺都能使用，要么会员只能在办卡店铺户提，注意~需要一个品牌一个账号，或者您做不同品类，可以共用

```js
{
  _id: String,              // 店铺_ID，文档ID
  merchantId: String,       // 关联 merchants._id
  storeName: String,        // 店铺名称，必填
  storeType: String,        // 店铺类型，可选，不填为null
  address: String,          // 地址，可选，不填为null
  storePhotos: Array,      // 店铺照片，可以多个，默认为[]
  businessLicense: String,  // 营业执照，Client端用，默认为null
  contactPhone: String,     // 联系电话，Client端用，默认为null
  contactWx: String,        // 联系微信，Client端用，默认为null
  status: Number,           // 0-关闭 1-正常营业 2-暂停营业 3-注销，默认为1，Client端用
  businessStartTime: String,     // 营业开始时间，Client端用，默认为null
  businessEndTime: String,       // 营业结束时间，Client端用，默认为null
  businessInfo: String,       // 营业信息文字描述，Client端用，默认为null
  openStoreTime: Date,        // 开店时间
  
  // 预设项目，办卡项，充值项，消费项，方便店员快速操作
  presetCardType: Array,  // 预设卡种项 [{memberType: Number, cardTypeName: String, value: Number}]，memberType: 1-余额+积分 2-积分 3-时长 4-次数，默认为[]
  presetRecharge: Array,  // 预设充值项 [{label: String, value: Number}]，支持余额，积分，时长（天），次数，默认为[]
  presetCost: Array,      // 预设消费项 [{label: String, value: Number}]，支持余额，积分，次数，默认为[]
  
  // ---- 跨店共享 ----
  shareGroup: Number,     // 跨店共享组标识，正整数。同值店铺共享会员，独立店铺为唯一值（新店铺默认同商家名下店铺shareGroup最大值递增分配，没开店从1开始）
  
  createTime: Date,
  updateTime: Date,
  extJson: Object
}
```

**索引字段：**
- `merchantId`（高频：查询某商家所有店铺）
- `status`

**字段变动：**
- **新增** `shareGroup` 跨店共享组标识，Number 类型。合并店铺时将目标店铺设为同组数值，解组时设为新的唯一数值（建议取当前商家所有店铺 shareGroup 最大值+1）
- **创建店铺时**，为商家所有店铺 shareGroup 最大值+1（确保不重复即可）
- **创建跨店组/加入跨店组时**，为目标跨店组 shareGroup中最小值（确保一致即可）
- **解散/退出跨店组时**，为商家所有店铺 shareGroup 最大值+1/+2/...（确保不重复即可）

> 员工账号由商家创建，账密直接交给员工使用。一个员工账号绑定一个店铺，无跨店场景。每个店铺最多支持创建50个员工账号。

```js

  merchantId: String,       // 所属商家 merchants._id
  storeId: String,          // 绑定店铺 stores._id（唯一，一账号一店铺）
  account: String,          // 登录账号（商家端创建时设置，在该商家下唯一）
  password: String,         // 登录密码（Argon2 哈希）
  name: String,             // 员工姓名
  role: String,             // 'manager'（店长）| 'employee'（店员）
  permission: Array,        // 权限列表
  remark: String,           // 备注

  tokenVersion: Number, // login\work令牌版本号（改密码/改权限时自增）

  status: Number,           // 0-离职（禁用） 1-在职（启用）
  entryTime: Date,          // 入职（创建）时间
  updateTime: Date
}
```

**索引字段：**
- `{ merchantId, account }`（唯一复合索引，账号在商家下唯一）
- `merchantId`
- `storeId`
- `status`

## 5. `member_cards` 集合

> 会员卡资产层，一个会员可在同商家名下持有多张卡（不同店铺各办一张）。唯一性约束：`{ memberId, storeId }` 联合唯一（同一会员在同一店铺只能持有一张卡）。

```js
{
  _id: String,              // 会员卡_ID，文档ID
  memberId: String,         // 所属会员 members._id
  merchantId: String,       // 冗余，所属商家 merchants._id
  phone: String,            // 手机号（明文存储，冗余，方便查询）
  memberName: String,       // 会员姓名备注，商家端使用，会员不可见
  memberAvatar: String,     // 会员头像，商家端验证本人用，可为 null

  storeId: String,          // 开卡店铺 stores._id

  // ---- 会员卡类型（创建时确定，不可修改）----
  memberType: Number,       // 1-余额+积分 2-积分 3-时长 4-次数

  // ---- 余额（memberType=1/4 时使用，次数卡存剩余次数）----
  balance: Number,          // 当前余额/剩余次数，默认 0

  // ---- 积分（memberType=1/2 时使用）----
  points: Number,           // 当前积分，默认 0
  cumulativePoints: Number, // 累计获取积分（只增不减）

  // ---- 时长（memberType=3 时使用）----
  expireTime: Date,         // 会员到期时间，非时长卡为 null

  // ---- 状态 ----
  status: Number,           // 1-正常 2-已过期 3-已冻结 4-已注销

  createTime: Date,
  updateTime: Date,
  extJson: Object
}
```

**索引字段：**
- `memberId`（查询会员名下所有卡）
- `merchantId`（查询商家所有会员卡）
- `storeId`（查询店铺所有会员卡）
- `memberType`
- `status`

**约束说明：**
- 同一会员在同店铺可多卡，业务层通过 `presetCardType` 控制开卡类型，避免重复开同类型卡

## 7. `transaction_records` 集合

> 时长卡的延期、日期变动，次数卡的充值/消费复用此集合。时长卡操作时 `balanceSnapshot` 置 null，次数卡 `amount` 为次数（正增负减）。

```js
{
  _id: String,
  memberCardId: String,     // 关联 member_cards._id（操作对象）
  memberId: String,         // 冗余，关联 members._id
  memberPhone: String,      // 冗余，方便展示
  merchantId: String,       // 冗余，方便查询某商家某天交易记录
  merberCardType: Number,   // 冗余，记录消费卡类型
  transactionType: Number,  // 1-充值 2-消费 3-退款 4-延期 5-日期变动
  amount: Number,           // 交易值（正入负出；共用，时长卡单位为天，次数卡单位为次，余额卡为元）
  balanceSnapshot: Number,  // 交易后余额快照，时长卡为 null
  operatorType: Number,     // 1-商家，2-店长，3-店员
  operatorId: String,       // 操作员 merchants._id 或 staffs._id
  transStoreId: String,     // 交易店铺 stores._id
  remark: String,           // 交易备注
  transactionTime: Date
}
```

**索引字段：**
- `memberCardId`
- `memberId`
- `merchantId`
- `transStoreId`
- `transactionType`
- `transactionTime`
- `{ memberCardId, transactionTime }`（会员卡交易历史分页）
- `{ merchantId, transactionTime }`（商家交易统计）

**跨店消费查询逻辑：**
```js
// 1. 通过手机号查会员
const member = await db.collection('members').where({ merchantId, phone }).get()

// 2. 查该会员在商家名下的所有卡
const cards = await db.collection('member_cards').where({ memberId: member._id }).get()

// 3. 获取当前店铺的 shareGroup
const currentStore = await db.collection('stores').doc(currentStoreId).get()
const currentShareGroup = currentStore.shareGroup

// 4. 获取所有同 shareGroup 的店铺ID列表
const sameGroupStores = await db.collection('stores')
  .where({ merchantId, shareGroup: currentShareGroup })
  .get()
const sameGroupStoreIds = sameGroupStores.data.map(s => s._id)

// 5. 筛选本店卡和跨店卡
const localCard = cards.data.find(c => c.storeId === currentStoreId)
const crossStoreCards = cards.data.filter(c => 
  c.storeId !== currentStoreId && sameGroupStoreIds.includes(c.storeId)
)

// 6. 返回给前端展示
return {
  localCard,      // 本店办理的卡（优先展示）
  crossStoreCards // 可跨店消费的卡
}
```

## 10. `reservation_templates` 集合

> 预约模板表

```js
{
  _id: String,
  storeId: String,
  merchantId: String,           // 冗余，方便按商家查询
  reservationTimeList: Array,   // 可预约时间段 ["08:00-09:00", ...]
  cancelRule: Array,            // 取消规则（对象数组），见下方格式说明
  advanceDays: Number,          // 可提前预约天数
  forbiddenDays: Array,         // 不支持预约的日期 ["周六", "2025-11-16"]
  customizeForbidden: Number,   // 0-不支持资源自定义禁止日期 1-支持
  effectiveStartTime: String,   // 'YYYY-MM-DD'
  effectiveEndTime: String,     // 'YYYY-MM-DD'，null 表示长期有效
  createTime: Date,
  updateTime: Date,
  extJson: Object
}
```

**`cancelRule` 格式说明：**
```js
cancelRule: [
  { minBefore: 30,  value: null, penaltyType: 'prohibit' }, // 30分钟内禁止取消
  { minBefore: 60,  value: 0.1,  penaltyType: 'ratio'    }, // 60分钟内取消扣10%
  { minBefore: 180, value: 5,    penaltyType: 'fixed'    }, // 180分钟内取消扣5元
]
// penaltyType: 'prohibit'（禁止，value=null）| 'ratio'（按比例，0<value<=1）| 'fixed'（固定金额，value>=0）
// 匹配逻辑：取 minBefore >= 剩余分钟数 中最小的规则；无匹配则免违约费
```

**索引字段：**
- `storeId`
- `merchantId`
- `{ storeId, effectiveStartTime, effectiveEndTime }`

**字段变动：**
- **新增** `merchantId` 冗余
- `effective_start_time / effective_end_time` 类型 `DATE` → 字符串 `'YYYY-MM-DD'`（同预约记录日期字段处理方式统一）
- `reservation_time_list / cancel_rule / forbidden_days` VARCHAR JSON字符串 → 直接存 `Array`
- `cancelRule` 格式由字符串数组 `['60:0.1', '180:5']` **升级为对象数组**（含 `penaltyType` 枚举，支持禁止取消场景）

```js
  createTime: Date,
  updateTime: Date
}
```

**索引字段：**
- `resourceId`
- `memberCardId`
- `memberId`
- `merchantId`
- `reservationDate`
- `reservationStatus`
- `{ resourceId, reservationDate, reservationStatus }`（判断某资源某天是否可预约）
- `{ memberCardId, reservationStatus, reservationDate }`（会员卡预约历史）

**字段变动：**
- `user_id` → `memberId`（关联 members 集合）
- `user_phone` → `memberPhone`（语义对齐）
- **新增** `merchantId` 冗余
- `transaction_id BIGINT` → `transactionId: String`
- `reservation_date DATE / start_time TIME / end_time TIME` → 字符串，云数据库不支持纯 DATE/TIME 类型
- **新增** `memberCardId` 关联 member_cards 集合

**防超卖策略（替代 Redisson 分布式锁，使用原子更新模拟乐观锁）：**

```js
// 步骤1：原子性检查该时段是否已存在有效预约
const { total } = await db.collection('reservation_records')
  .where({
    resourceId: targetResourceId,
    reservationDate: targetDate,      // 'YYYY-MM-DD'
    startTime: targetStartTime,       // 'HH:mm'
    reservationStatus: _.in([0, 1])   // 0-待使用 1-已使用
  })
  .count()

// 步骤2：count > 0 直接拒绝，count === 0 才允许写入
if (total > 0) throw new Error('TIME_SLOT_OCCUPIED')

// 步骤3：写入预约记录
// { resourceId, reservationDate, startTime } 设有唯一复合索引
// 极端并发时仅第一条写入成功，其余因唯一索引冲突失败，返回"预约失败，请重试"
await db.collection('reservation_records').add({ ... })
```

> 步骤1的 count 检查处理99%场景；唯一复合索引是最后一道并发兜底，两者共同保证零超卖。

**索引补充：**
- `{ resourceId, reservationDate, startTime }`（**唯一复合索引**，防并发超卖）

## 13. `verify_codes` 集合

> 短信验证码存储，替代 Redis Hash + SET EX 结构。含1分钟频率限制、5分钟有效期与每日5次上限。

```js
{
  _id: String,
  phone: String,          // 手机号（明文存储，传输时由前端 AES 加密，云函数解密后存入）
  code: String,           // 6位数字验证码（明文，集合不对外暴露）
  codeExpireTime: Date,   // createTime + 5分钟（验证码过期时间，但是当有新验证码请求时，更新expireTime）
  used: Boolean,          // 单次使用，使用(注册/找回密码)后置 true，防重复消费
  dailyCount: Number,     // 当天已发送次数（上限5次）
  lastSendTime: Date,     // 最近有效调用时间（1分钟频率限制，每次成功发生验证码更新）
  expireTime: Date       // 次日零点（清理时间）
}
```

**索引字段：**
- `phone` 查询指定手机号最新记录
- `expireTime` 验证码过期时间

**核心设计逻辑：**
- 每天凌晨，定时清理expireTime
- 通过phone查询，每日首次不在集合中，则新增一条记录
- 非首次调用，phone能查到在集合中有记录，如果请求时间戳 < lastSendTime+1分钟，则返回请求频繁，并返回剩余等待秒数；如果请求时间戳 >= lastSendTime+1分钟，检测已使用次数，超过5次返回今日次数用尽，否则是成功调用，更新code、expireTime、lastSendTime，并增加dailyCount，更新

## 14. `token_blacklist` 集合

> 单令牌级别注销，与 `tokenVersion` 全局失效互补。
> `tokenVersion` 自增 = 退出所有设备/工作令牌失效；黑名单 = 退出单设备。
> 使用场景，用户主动退出登录时，将自动登录令牌、访管令牌、工作令牌加入黑名单；刷新令牌时，将旧令牌加入黑名单；

```js
{
  _id: String,
  jti: String,          // JWT jti 字段，令牌唯一标识
  expireTime: Date     // 与原令牌 exp 一致，过期后文档无意义
}
```

**索引字段：**
- `jti`（唯一索引，每次请求校验）
- `expireTime`（定时清理，每天凌晨）

**校验顺序（JwtUtils 层）：**
1. （除注册全通用校验）JWT 签名 + 过期校验
2. （仅登录接口，工作接口使用）查询 `token_blacklist`，`jti` 存在 → 401 Token Revoked
3. （仅）tokenVersion版本校验
备注： JWT层不做，业务层做校验，可以提供JWT工具，但是这个版本校验是可选（剥离处理），例如只对写操作校验，读校验跳过版本校验，可以加快响应速度，减轻数据库负担。

## 15. `idempotent_keys` 集合

> 幂等防重，替代 Redis `SET NX EX`。防止充值、消费、退款等关键接口重复提交。记录存活 3 分钟。

```js
{
  _id: String,
  requestId: String,    // 客户端生成的唯一请求ID（UUID）
  uid: String,          // 发起请求的用户 _id（merchants / staffs）
  expireTime: Date,     // createTime + 3分钟
  createTime: Date
}
```

**索引字段：**
- `requestId`（唯一索引，防重核心），uid+requestId 绝对唯一
- `expireTime`（定时清理，每5分钟）

**使用方式：**
- 客户端每次操作生成 `requestId`（UUID）随请求上传
- 云函数尝试写入，唯一索引冲突 → 返回"请求已处理，请勿重复提交"
- 写入成功 → 正常执行业务逻辑，请求结束后无需主动删除

## 16. `login_fail_records` 集合

> 登录失败锁定，替代 Redis INCR + SET EX 结构。梯度锁定策略防暴力破解。

```js
{
  _id: String,
  role: String,         // 'merchant' | 'staff' | 'member'
  identifier: String,   // 锁定标识：①merchant: 手机号明文; ②staff：账号名（生成时唯一）; ③member: 手机号明文
  failCount: Number,    // 当前连续失败次数，如果成功，则直接删除这条记录
  lockUntil: Date,      // 锁定到期时间，null 表示未锁定
  lastFailTime: Date,
  updateTime: Date
}
```

**索引字段：**
- `identifier`（唯一索引）
- 清理频率：次日凌晨清理

**梯度锁定策略：**

| 连续失败次数 | 锁定时长 |
|---|---|
| 5次 | 15 分钟 |
| 再失败5次 | 60 分钟 |
| 再失败5次 | 24 小时 |
