# 员工管理模块 API 接口设计文档（新版）

## 文档版本
- **版本号**: v2.0
- **创建日期**: 2025-10-30
- **适用系统**: 会员管理系统
- **设计原则**: 基于工作令牌机制，支持商家/店长分级管理

---

## 一、权限模型说明

### 1.1 角色定义
| 角色 | 数据库值 | API响应值 | 说明 |
|------|----------|-----------|------|
| 商家 | - | MERCHANT | 商户所有者，拥有全部权限 |
| 店长 | manager | STORE_MANAGER | 店铺管理员，拥有部分管理权限 |
| 店员 | employee | STAFF | 普通员工，权限受限 |

### 1.2 权限字段枚举
```json
{
  "manager": [
    "staff_add"  // 员工添加（店长默认拥有）
  ],
  "employee": [
    "member_card_create",      // 会员卡创建（店员默认不拥有）
    "transaction_recharge",    // 会员卡充值（店员默认不拥有）
    "reservation_disable",     // 停用预约资源（店员默认不拥有）
    "reservation_occupy"       // 线下占用预约资源（店员默认不拥有）
  ]
}
```

**说明**：
- 商家拥有全部权限，无需校验 `permission` 字段
- 店长/店员的权限存储在 `t_work_relation.permission` 字段（JSON格式）
- 工作令牌中包含权限列表，直接从令牌读取，无需查询数据库

### 1.3 工作令牌（Work Token）

#### 工作令牌 Payload 结构
| 字段名 | 类型 | 说明 |
|--------|------|------|
| user_id | String | 用户ID（UUID） |
| role | String | 角色：manager/employee |
| merchant_id | String | 所属商家ID |
| store_id | String | 工作店铺ID |
| permissions | Array | 权限列表 |
| token_version | Integer | 工作令牌版本号 |
| device_id | String | 设备ID |
| login_ip | String | 登录IP |
| token_type | Integer | 令牌类型：3-工作令牌 |
| exp | Long | 过期时间戳 |
| jti | String | 令牌唯一ID |

#### 工作令牌申请接口
- **URL**: `/api/v1/tokens/work`
- **方法**: POST
- **请求参数**:
  ```json
  {
    "storeId": "uuid-xxx"  // 工作店铺ID
  }
  ```
- **响应**:
  ```json
  {
    "code": 0,
    "message": "工作令牌申请成功",
    "data": {
      "token": "Bearer eyJhbGci...",
      "expireTime": "2025-10-30 12:00:00",
      "jti": "uuid-xxx",
      "storeId": "uuid-xxx",
      "merchantId": "uuid-xxx",
      "role": "manager",
      "permissions": ["staff_add", "member_card_create"],
      "tokenVersion": 1
    }
  }
  ```

**注意**：
1. 权限列表自动从数据库 `t_work_relation.permission` 读取，前端无需传入
2. 令牌版本号从数据库 `t_work_relation.token_version` 读取
3. 商家可申请任一店铺的工作令牌，店长/店员只能申请有工作关系的店铺令牌

### 1.4 工作令牌版本管理

#### 令牌版本号递增场景
| 场景 | 操作 | 影响 |
|------|------|------|
| 员工角色调整 | `token_version++` | 旧工作令牌失效，需重新申请 |
| 员工权限修改 | `token_version++` | 旧工作令牌失效，需重新申请 |
| 员工解绑 | 删除工作关系记录 | 查询不到记录，拒绝并提示"你已不是本店员工" |

#### 版本号校验流程
```
客户端请求 → 提取令牌中的 token_version
           ↓
查询数据库 t_work_relation.token_version
           ↓
版本号不一致 → 返回错误 10007（令牌版本过期）
           ↓
版本号一致 → 继续处理请求
```

## 二、数据库设计

### 2.1 t_work_relation 表结构（新增字段）

| 字段名 | 类型 | 是否为空 | 备注 | 索引 |
|--------|------|----------|------|------|
| id | BIGINT | NOT NULL | 关系ID，主键（自增） | PRIMARY KEY |
| store_id | BINARY(16) | NOT NULL | 店铺ID（外键） | INDEX, FOREIGN KEY |
| **merchant_id** | **BINARY(16)** | **NOT NULL** | **所属商家ID（冗余字段）** | **INDEX** |
| user_id | BINARY(16) | NOT NULL | 用户ID（外键） | INDEX, FOREIGN KEY |
| role | VARCHAR(20) | NOT NULL | 角色：manager/employee | INDEX |
| name | VARCHAR(50) | NULL | 员工姓名 | - |
| permission | JSON | NOT NULL | 权限列表（JSON格式） | - |
| **token_version** | **INT** | **NOT NULL DEFAULT 1** | **工作令牌版本号** | **-** |
| remark | VARCHAR(100) | DEFAULT '' | 备注 | - |
| status | TINYINT | DEFAULT 1 | 状态：0-离职，1-在职 | - |
| entry_time | DATETIME | NOT NULL | 入职时间 | - |
| update_time | DATETIME | NOT NULL | 更新时间 | - |
| - | - | - | 联合唯一约束：store_id + user_id | UNIQUE KEY |

**新增字段说明**：
1. **merchant_id**: 冗余字段，减少查询时的关联操作，便于跨店铺批量查询员工
2. **token_version**: 工作令牌版本号，每次角色/权限修改时递增，确保令牌实时有效

### 3.2 接口1：员工添加（绑定）

#### 3.2.1 基础信息
- **业务背景**: 商家/店长为店铺添加员工，通过手机号和邀请码验证身份
- **权限要求**: 
  - 商家：可使用普通令牌或工作令牌
  - 店长：必须使用工作令牌，且工作令牌的 `permissions` 中包含 `"staff_add"`
- **URL**: `/api/v1/staffs`
- **HTTP方法**: POST

#### 3.2.2 请求参数

**请求头（Header）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {令牌} |

**请求体（Body）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 条件 | 店铺ID，商家使用普通令牌时必填 |
| staffPhone | String | 是 | 待添加员工的手机号（明文，11位） |
| staffInviteCode | String | 是 | 待添加员工的邀请码 |
| staffRole | String | 是 | 员工角色：STAFF-普通员工，STORE_MANAGER-店长 |
| staffPermission | String | 否 | 员工权限配置（JSON字符串，可选）|
| staffName | String | 否 | 员工姓名（最多20位，可选） |
| remark | String | 否 | 备注（最多100位，可选） |

**staffPermission 示例**：
```json
{"manager": ["staff_add"]}
// 或
{"employee": ["member_card_create", "transaction_recharge"]}
// 空权限
{"employee": []}
```

#### 3.2.3 响应参数
| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Int | 状态码：200-成功，其他-失败 |
| message | String | 响应消息 |
| data.staffId | String | 员工的userId（UUID） |
| data.storeId | String | 关联店铺ID |
| data.relationId | Long | 工作关系ID |
| data.tokenVersion | Integer | 工作令牌初始版本号 |
| data.createTime | String | 添加时间（yyyy-MM-dd HH:mm:ss） |

#### 3.2.4 响应示例
```json
{
  "code": 200,
  "message": "员工添加成功",
  "data": {
    "staffId": "uuid-1234-90ab-cdef-5678",
    "storeId": "uuid-cdef-1234-5678-90ab",
    "relationId": 1,
    "tokenVersion": 1,
    "createTime": "2025-10-30 09:30:00"
  }
}
```

#### 3.2.5 业务规则
1. 验证手机号已注册且邀请码正确
2. 检查该用户是否已绑定该店铺（唯一约束：store_id + user_id）
3. 商家使用普通令牌时，需要验证店铺归属
4. 店长使用工作令牌时，自动从令牌提取 `storeId`，且需验证 `permissions` 包含 `"staff_add"`
5. 将用户的 `user_type` 设置为 3（员工用户）
6. 初始化 `token_version=1`
7. 设置 `merchant_id`（从令牌或店铺关联查询）

#### 3.2.6 可能的错误码
- 90001: 参数错误
- 20003: 用户不存在
- 20004: 邀请码错误
- 50001: 员工已关联该店铺
- 10004: 令牌权限不足（店长无 staff_add 权限）

### 3.3 接口2：员工信息查询（单个）

#### 3.3.1 基础信息
- **业务背景**: 商家/店长查询员工详情（含权限），员工查询自身基础信息（不含权限）

- **URL**: `/api/v1/staffs/{staffId}`

#### 3.3.2 请求参数

**路径参数**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| staffId | String | 是 | 员工ID（userId） |

**请求头（Header）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {令牌} |

**查询参数（Query）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 条件 | 店铺ID，商家使用普通令牌时必填，员工自查时也必填 |

#### 3.3.3 响应参数（商家/店长查询）

|--------|------|------|

| data.staffRole | String | 员工角色：STAFF/STORE_MANAGER |

### 3.4 接口3：员工信息查询（列表）

#### 3.4.1 基础信息
- **业务背景**: 商家/店长分页查询指定店铺的员工列表，支持筛选
- **权限要求**: 商家/店长
- **URL**: `/api/v1/staffs`
- **HTTP方法**: GET

#### 3.4.2 请求参数

**请求头（Header）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {令牌} |

**查询参数（Query）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 条件 | 店铺ID，商家使用普通令牌时必填 |
| staffRole | String | 否 | 角色筛选：STAFF/STORE_MANAGER |
| staffName | String | 否 | 姓名模糊筛选 |
| pageNum | Int | 否 | 页码（从1开始，默认1） |
| pageSize | Int | 否 | 每页条数（1-50，默认10） |

#### 3.4.3 响应参数
| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Int | 状态码：200-成功 |
| message | String | 响应消息 |
| data.total | Long | 总员工数量 |
| data.pageNum | Int | 当前页码 |
| data.pageSize | Int | 每页条数 |

| data.list[].staffName | String | 员工姓名 |

| data.list[].staffRole | String | 员工角色 |
| data.list[].entryTime | String | 入职时间（yyyy-MM-dd HH:mm:ss） |

**默认排序**：按 `entry_time` 降序（最新入职的在前）

### 3.6 接口5：员工角色调整

#### 3.6.1 基础信息
- **业务背景**: 商家调整员工角色（普通员工 ↔ 店长），需要管理令牌
- **权限要求**: 仅商家，需要管理令牌（通过二级密码获取）
- **URL**: `/api/v1/staffs/{staffId}/role`
- **HTTP方法**: PUT

#### 3.6.2 请求参数

**路径参数**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| staffId | String | 是 | 员工ID |

**请求头（Header）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {管理令牌} |

**请求体（Body）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| merchantId | String | 是 | 商家ID（与令牌一致） |
| storeId | String | 是 | 员工关联店铺ID |
| targetRole | String | 是 | 目标角色：STAFF/STORE_MANAGER |

#### 3.6.3 响应参数
| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Int | 状态码：200-成功 |
| message | String | 响应消息 |
| data.staffId | String | 员工ID |
| data.oldRole | String | 调整前角色 |
| data.newRole | String | 调整后角色 |
| data.adjustTime | String | 调整时间（yyyy-MM-dd HH:mm:ss） |

#### 3.6.4 响应示例
```json
{
  "code": 200,
  "message": "员工角色调整成功",
  "data": {
    "staffId": "uuid-1234-90ab-cdef-5678",
    "oldRole": "STAFF",
    "newRole": "STORE_MANAGER",
    "adjustTime": "2025-10-30 14:00:00"
  }
}
```

#### 3.6.5 业务规则
1. 验证管理令牌有效性（令牌类型必须为4-管理令牌）
2. 检查目标角色是否与当前角色相同
3. **角色调整后，`token_version++`，旧工作令牌失效**
4. 更新 `update_time`

### 3.7 接口6：员工权限修改

#### 3.7.1 基础信息
- **业务背景**: 商家可修改任意员工权限，店长只能修改店员权限（不能修改店长权限）
- **权限要求**: 
  - 商家：可使用普通令牌或工作令牌，可修改任意员工
  - 店长：必须使用工作令牌，只能修改 `role=employee` 的员工
- **URL**: `/api/v1/staffs/{staffId}/permission`
- **HTTP方法**: PUT

#### 3.7.2 请求参数

**路径参数**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| staffId | String | 是 | 员工ID |

**请求头（Header）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {令牌} |

**请求体（Body）**
| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 条件 | 店铺ID，商家使用普通令牌时必填 |
| newPermission | String | 是 | 新权限配置（JSON字符串） |

**newPermission 示例（店长）**：


**newPermission 示例（店员）**：
```json

```

**空权限示例**：


| code | Int | 状态码：200-成功 |
| message | String | 响应消息 |
| data.updateTime | String | 修改时间（yyyy-MM-dd HH:mm:ss） |

#### 3.7.4 响应示例
```json
{
  "code": 200,
  "message": "员工权限修改成功",
  "data": {
    "updateTime": "2025-10-30 15:00:00"
  }

```

#### 3.7.5 业务规则
1. 商家可修改任意员工权限
2. 店长只能修改 `role=employee` 的员工权限，不能修改店长权限
3. **权限修改后，`token_version++`，旧工作令牌失效**
4. 更新 `update_time`

### 3.8 接口7：员工解绑（注销）

#### 3.8.1 基础信息
- **业务背景**: 商家解除员工与店铺的绑定关系，永久删除工作关系记录
- **权限要求**: 仅商家，需要管理令牌（通过二级密码获取）
- **URL**: `/api/v1/staffs/{staffId}/unbind`
- **HTTP方法**: PUT

#### 3.8.2 请求参数

**路径参数**

1. 验证管理令牌有效性（令牌类型必须为4-管理令牌）

## 四、权限校验流程

### 4.1 商家/店长身份识别

#### 从令牌类型识别
```
if (令牌类型 == 普通令牌) {
    角色 = 商家（需验证 user_type=2）
    storeId = 从请求参数获取（必填）
} else if (令牌类型 == 工作令牌) {
    角色 = 从令牌.role 获取（manager/employee → STORE_MANAGER/STAFF）
    storeId = 从令牌.store_id 获取
    merchantId = 从令牌.merchant_id 获取
}
```

### 4.2 店长权限校验

#### 校验流程
```
1. 检查令牌类型是否为工作令牌
2. 检查令牌中的 role 是否为 "manager"
3. 检查令牌中的 permissions 是否包含所需权限
   例：员工添加需要 "staff_add"
4. 检查令牌版本号是否与数据库一致
5. 检查目标员工的 storeId 是否与令牌中的 store_id 一致
```

### 4.3 工作令牌版本号校验

#### 校验伪代码
```java
// 提取令牌中的版本号
Integer tokenVersion = jwtUtils.extractTokenVersion(workToken);

// 查询数据库中的版本号
WorkRelation relation = workRelationRepository
    .findByStoreIdAndUserIdAndStatus(storeIdBytes, userIdBytes, 1)
    .orElseThrow(() -> new BusinessException(50002, "你已不是本店员工"));

// 版本号对比
if (!tokenVersion.equals(relation.getTokenVersion())) {
    throw new BusinessException(10007, "工作令牌版本过期，请重新申请");
}
```

### 6.1 令牌版本号递增触发点

### 7.2 权限解析工具方法

```java
/**
 * 从JSON权限字段解析权限列表
 * 
 * @param permissionJson 权限JSON字符串
 * @param role 角色：manager/employee
 * @return 权限列表
 */
private List<String> parsePermissionsFromJson(String permissionJson, String role) {
    try {
        JSONObject json = new JSONObject(permissionJson);
        JSONArray array;
        
        if ("manager".equals(role)) {
            array = json.optJSONArray("manager");
        } else {
            array = json.optJSONArray("employee");
        }
        
        if (array == null) {
            return Collections.emptyList();
        }
        
        List<String> permissions = new ArrayList<>();
        for (int i = 0; i < array.length(); i++) {
            permissions.add(array.getString(i));
        }
        return permissions;
    } catch (Exception e) {
        log.error("解析权限JSON失败: {}", permissionJson, e);
        return Collections.emptyList();
    }
}
```

```java
}
```
