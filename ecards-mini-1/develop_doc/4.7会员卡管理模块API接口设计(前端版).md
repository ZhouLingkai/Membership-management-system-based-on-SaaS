### 3.2 接口1：会员卡办理（手机号快速办理）

#### 3.2.1 基础信息

- **业务背景**: 商家/店长/店员通过手机号为用户办卡，用户可能尚未注册，办理的会员卡状态为"未激活"，待用户注册后可激活
- **权限要求**: 商家/店长/店员（工作令牌）
- **URL**: `/api/v1/member-cards/create-by-phone`
- **HTTP方法**: POST

#### 3.2.2 请求参数

**请求头（Header）**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {工作令牌或普通令牌} |

**请求体（Body）**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 是 | 店铺ID（UUID格式） |
| cardTypeId | Long | 是 | 卡种ID |
| memberPhone | String | 是 | 会员手机号（AES256CBC加密或明文11位） |
| memberName | String | 否 | 会员预留姓名（最多64位） |
| initialBalance | BigDecimal | 否 | 初始余额（余额卡适用，默认0.00） |
| initialTimes | Integer | 否 | 初始次数（次数卡适用，默认0） |
| initialPoints | Integer | 否 | 初始积分（默认0） |
| expireTime | String | 否 | 到期时间（ISO8601格式，不填则默认60年后） |

#### 3.2.3 响应参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Int | 状态码：200-成功，其他-失败 |
| message | String | 响应消息 |
| data.memberCardId | String | 会员卡ID（UUID格式） |
| data.cardTypeId | Long | 卡种ID |
| data.cardTypeName | String | 卡种名称 |
| data.cardTtype | Integer | 卡种类型：1-余额卡，2-次数卡，3-时效卡，4-积分卡 |
| data.cardTtypeName | String | 卡种类型名称 |
| data.memberPhone | String | 会员手机号（明文） |
| data.memberName | String | 会员预留姓名 |
| data.balance | BigDecimal | 余额 |
| data.times | Integer | 剩余次数 |
| data.points | Integer | 积分 |
| data.status | Integer | 会员卡状态：0-未激活，1-正常 |
| data.statusName | String | 状态名称 |
| data.openCardTime | String | 开卡时间（ISO8601格式） |
| data.expireTime | String | 到期时间（ISO8601格式） |
| data.message | String | 提示信息（如：该手机号尚未注册，会员卡将处于"未激活"状态） |

#### 3.2.4 响应示例

```json
{
  "code": 200,
  "message": "会员卡办理成功",
  "data": {
    "memberCardId": "uuid-1234-5678-90ab-cdef",
    "cardTypeId": 1001,
    "cardTypeName": "黄金会员卡",
    "cardTtype": 1,
    "cardTtypeName": "余额卡",
    "memberPhone": "13800138000",
    "memberName": "张三",
    "balance": 100.00,
    "times": 0,
    "points": 0,
    "status": 0,
    "statusName": "未激活",
    "openCardTime": "2025-11-02T10:30:00",
    "expireTime": "2085-11-02T10:30:00",
    "message": "该手机号尚未注册，会员卡将处于"未激活"状态"
  }
}
```

#### 3.2.5 业务规则

1. **令牌验证**：
   - 如果是工作令牌（token_type=3）：验证工作令牌有效性，验证工作店铺（token.store_id = storeId）
   - 如果是普通令牌（token_type=1）且商家（user_type=2）：验证店铺归属（store.merchant_id = token.merchant_id）
   - **店员权限验证**：如果是工作令牌且角色为STAFF，需验证工作关系中是否有`member_card_create`权限
2. 验证卡种存在且属于指定店铺
3. 验证商家认证状态：必须为1（已认证）或2（未认证测试中）
4. 验证商家会员数量限制：
   - 普通商家（merchant_level=1）：名下最多200个会员卡（**不用去重，只统计会员卡个数**）
   - VIP/SVIP商家：不限制
5. **强制解密手机号**，验证手机号格式（必须为11位手机号）
6. 检查该手机号在该卡种下是否已办卡（避免重复办理）
7. 自动填充冗余字段：store_id、merchant_id、card_ttype
8. **按手机号查询该用户是否注册**，若未注册，会员卡状态设置为0（未激活），user_id设置为NULL（待激活后关联）；若已注册，会员卡状态设置为1（正常），user_id设置为对应的user_id。
9. 办卡信息记录到t_registration_card_record表

#### 3.2.6 参数校验规则

| 参数名 | 校验规则 | 错误提示 |
|--------|----------|----------|
| storeId | 非空、UUID格式、店铺存在 | "店铺ID不能为空"、"店铺ID格式错误"、"店铺不存在" |
| cardTypeId | 非空、卡种存在、卡种属于店铺 | "卡种ID不能为空"、"卡种不存在"、"卡种与店铺不匹配" |
| memberPhone | 非空、AES解密成功、手机号格式正确 | "手机号不能为空"、"手机号格式错误" |
| memberName | 最多30位 | "会员姓名长度不能超过30位" |
| initialBalance | ≥0 | "初始余额不能为负数" |
| initialTimes | ≥0 | "初始次数不能为负数" |
| initialPoints | ≥0 | "初始积分不能为负数" |

#### 3.2.7 可能的错误码

- 90001: 参数错误
- 30002: 商家信息不存在
- 40001: 非商家用户，无权限操作
- 40002: 店铺不存在
- 60002: 卡种不存在
- 60003: 卡种与店铺不匹配
- 70001: 该手机号已办理该卡种
- 70002: 商家会员数量已达上限（普通商家最多200个会员，请开通特权）
- 10004: 令牌权限不足
- 10007: 工作令牌版本过期

### 3.3 接口2：会员卡办理（线下扫码办理）

#### 3.3.1 基础信息

- **业务背景**: 商家/店长/店员扫描用户的特权二维码，前端获取用户特权令牌后直接传递给后端，后端验证一次性有效性并解析令牌提取userId，办理后状态为"正常"
- **权限要求**: 商家/店长/店员（工作令牌）
- **URL**: `/api/v1/member-cards/create-by-scan`
- **HTTP方法**: POST
- **实现状态**: ✅ 已实现

#### 3.3.2 请求参数

**请求头（Header）**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| Authorization | String | 是 | Bearer {工作令牌或普通令牌} |

**请求体（Body）**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| storeId | String | 是 | 店铺ID（UUID格式） |
| cardTypeId | Long | 是 | 卡种ID |
| privilegeToken | String | 是 | 用户特权令牌（从二维码中获取，5分钟有效，一次性） |
| memberName | String | 否 | 会员预留姓名（最多64位） |
| initialBalance | BigDecimal | 否 | 初始余额（余额卡适用，默认0.00） |
| initialTimes | Integer | 否 | 初始次数（次数卡适用，默认0） |
| initialPoints | Integer | 否 | 初始积分（默认0） |
| expireTime | String | 否 | 到期时间（ISO8601格式，不填则默认60年后） |

#### 3.3.3 响应参数

| 参数名 | 类型 | 说明 |
|--------|------|------|
| code | Int | 状态码：200-成功，其他-失败 |
| message | String | 响应消息 |
| data.memberCardId | String | 会员卡ID |
| data.cardTypeId | Long | 卡种ID |
| data.cardTypeName | String | 卡种名称 |
| data.userId | String | 用户ID |
| data.memberName | String | 会员预留姓名 |
| data.memberPhone | String | 会员手机号 |
| data.status | Integer | 会员卡状态：1-正常 |
| data.statusName | String | 状态名称 |
| data.openCardTime | String | 开卡时间 |
| data.activateTime | String | 激活时间 |

#### 3.3.4 响应示例

```json
{
  "code": 200,
  "message": "会员卡办理成功",
  "data": {
    "memberCardId": "uuid-1234-5678-90ab-cdef",
    "cardTypeId": 1001,
    "cardTypeName": "黄金会员卡",
    "userId": "uuid-5678-90ab-cdef-1234",
    "memberName": "李四",
    "memberPhone": "encrypted_phone_string",
    "status": 1,
    "statusName": "正常",
    "openCardTime": "2025-11-02 10:30:00",
    "activateTime": "2025-11-02 10:30:00"
  }
}
```

#### 3.3.5 业务规则

1. **工作令牌验证**（操作者）：
   - 如果是工作令牌（token_type=3）：验证工作令牌有效性，验证工作店铺（token.store_id = storeId）
   - 如果是普通令牌（token_type=1）且商家（user_type=2）：验证店铺归属（store.merchant_id = token.merchant_id）
   - **店员权限验证**：如果是工作令牌且角色为STAFF，需验证工作关系中是否有`member_card_create`权限

2. **特权令牌验证**（目标用户）【已实现】：
   - 解析 `privilegeToken`，提取 `user_id` 和 `jti`
   - 检查 Redis 黑名单：如果存在 `blacklist:{jti}`，返回错误 70011 "该二维码已使用，请重新生成"
   - 验证令牌类型：`single_use` 必须为 `true`
   - 验证权限：`permissions` 必须包含 `CARD_CREATE`
   - 验证令牌未过期（exp > 当前时间）
   - **复用现有黑名单机制**：验证通过后，执行办卡操作完成时，立即将 `jti` 加入 Redis 黑名单，TTL 设置为令牌剩余有效期（最多300秒）

3. 验证卡种存在且属于指定店铺
4. 验证商家认证状态：必须为1（已认证）或2（未认证测试中）
5. 验证商家会员数量限制（同接口1）
6. 验证从特权令牌中提取的 `user_id` 存在且为有效用户
7. 检查该用户在该卡种下是否已办卡（避免重复办理）
8. 从用户表查询手机号并存储到member_phone
9. 会员卡状态设置为1（正常）
10. activate_time设置为当前时间
11. user_id设置为从特权令牌提取的userId
12. 自动填充冗余字段
13. 办卡信息记录到t_registration_card_record表
14. **立即将特权令牌jti加入Redis黑名单**（防止重复使用）

#### 3.3.6 可能的错误码

- 90001: 参数错误
- 20003: 用户信息不存在
- 30002: 商家信息不存在
- 40002: 店铺不存在
- 60002: 卡种不存在
- 60003: 卡种与店铺不匹配
- 70003: 该用户已办理该卡种
- 70002: 商家会员数量已达上限
- 70011: 该二维码已使用，请重新生成
- 10004: 令牌权限不足
- 10012: 特权令牌无效或已过期
- 10013: 特权令牌权限不足（非办卡权限）

### 3.5 接口4：查询店铺会员卡列表

#### 3.11.5 业务规则

**验证阶段**：

1. **工作令牌验证**（操作者）：
   - 如果是工作令牌（token_type=3）：验证工作令牌有效性，验证工作店铺（token.store_id = storeId）
   - 如果是普通令牌（token_type=1）且商家（user_type=2）：验证店铺归属（store.merchant_id = token.merchant_id）

2. **特权令牌验证**（目标用户）【依赖JWT分层令牌机制】：
   - 解析 `privilegeToken`，提取 `user_id` 和 `jti`
   - 检查 Redis 黑名单：如果存在 `blacklist:{jti}`，返回错误 70011 "该二维码已使用，请重新生成"
   - 验证令牌类型：`single_use` 必须为 `true`
   - 验证权限：`permissions` 必须包含 `QUERY_MCARD`
   - 验证令牌未过期（exp > 当前时间）
   - **复用现有黑名单机制**：验证通过后，执行查询操作完成时，立即将 `jti` 加入 Redis 黑名单，TTL 设置为令牌剩余有效期（最多300秒）

3. 验证商家/归属商家认证状态：必须为1（已认证）或2（未认证测试中）

**查询阶段**：

4. **根据memberCardId有无决定查询逻辑**：

   **场景A：有memberCardId（会员卡详情查询）**
   - 验证会员卡存在
   - 验证 `card.user_id = privilegeToken.user_id`（防止memberCardId字段造假）
   - 验证会员卡归属于当前商家（card.merchant_id = 商家ID）
   - 返回完整的会员卡详情（参考接口9）

   **场景B：无memberCardId（会员卡列表查询）**
   - 从特权令牌提取的user_id查询该用户在该店铺可用的会员卡
   - **本店卡查询**：user_id = privilegeToken.user_id AND store_id = storeId
   - **跨店卡查询**：user_id = privilegeToken.user_id AND merchant_id = 商家ID AND store_id != storeId AND card_type.cross_store = 1
   - 返回所有可用卡（不分页，上限500条）
   - 不能越界查询其他店铺/商家的会员卡

5. **查询完成后立即将特权令牌jti加入Redis黑名单**（防止重复使用）
